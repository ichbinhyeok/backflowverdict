# 04 Public Support Model

## 1) Active product posture
BackflowPath is an owner-facing utility-first compliance site with secondary public provider directories and a manual request-help workflow.

## 2) Public provider model
- Provider profiles may appear on utility, tester, metro, and provider routes only as clearly labeled public records.
- Official authority lists always outrank non-official directories.
- A provider profile should never imply approval, endorsement, or authority status without source support.
- Provider browse is optional next-step support, not the main answer.

## 3) Request-help model
- The request form exists so a user can ask for help after seeing the governing utility rule.
- Requests are stored for manual review.
- Manual follow-up may point the user back to the official workflow or toward a matching public provider profile.
- The request form is not an instant dispatch tool and should not imply automatic matching.

## 4) Directory CTA rules
- Utility page -> official source first, request help and public directories second
- Approved-testers page -> official list first, public provider profiles separate below it
- Find-a-tester page -> explicit non-official directory framing
- Failed-test page -> repair/retest sequence first, public provider options after the workflow is clear
- Provider pages -> contact and authority-source clarity, not promotional positioning

## 5) Consent and privacy rules
- Do not imply that a checked box means immediate outreach.
- Store source page and page intent for every request.
- Explain that requests are reviewed manually under the public privacy notice.
- Keep request review separate from the official rule display.

## 6) Product constraints
- Do not add private setup, internal dashboards, or unpublished routing artifacts back into the public product without a new strategy decision.
- If monetization is revisited later, it must preserve the same trust boundary: official workflow first, public support surfaces second.
